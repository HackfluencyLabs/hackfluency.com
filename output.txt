El problema no es Lighthouse, porque claramente sí genera los archivos. El problema es el path desde donde upload-artifact intenta subirlos.

Tu propio log muestra esto:

📂 Listing .lighthouseci directory:
total 4180
...
lhr-1773505821129.json

pero en el step siguiente:

Warning: No files were found with the provided path: .lighthouseci/

Eso casi siempre ocurre por una de estas 3 razones en GitHub Actions.

1️⃣ El path relativo no coincide con el working directory

Tu workflow ejecuta todo desde:

/home/runner/work/hackfluency.com/hackfluency.com

pero upload-artifact busca relativo a github.workspace.

Normalmente coincide, pero a veces cuando hay download-artifact, tar, etc., el path efectivo cambia.

Solución segura

Usa path absoluto basado en workspace:

- name: Upload Lighthouse Reports
  if: always()
  uses: actions/upload-artifact@v4
  with:
    name: lighthouse-reports-${{ github.run_number }}
    path: ${{ github.workspace }}/.lighthouseci
    retention-days: 30
2️⃣ .lighthouseci a veces queda vacío al momento del upload

LHCI escribe archivos asíncronamente al terminar el proceso.

Aunque tú ya listaste el directorio, a veces el runner llega al step antes de flush final.

Solución robusta:

- name: Wait for Lighthouse reports
  run: |
    sleep 2
    ls -la .lighthouseci
3️⃣ upload-artifact@v4 requiere glob cuando es directorio

Hay un cambio en v4: subir directorios a veces falla si no se usa wildcard.

Solución recomendada:

path: .lighthouseci/**
🔧 Fix mínimo (recomendado)

Cambia tu step por esto:

- name: Upload Lighthouse Reports
  if: always()
  uses: actions/upload-artifact@v4
  with:
    name: lighthouse-reports-${{ github.run_number }}
    path: ${{ github.workspace }}/.lighthouseci/**
    retention-days: 30
🧠 Mejora adicional (muy útil)

Si quieres ver los reportes directo en Actions UI:

- name: Upload Lighthouse HTML
  if: always()
  uses: actions/upload-artifact@v4
  with:
    name: lighthouse-html
    path: .lighthouseci/*.html

Así puedes abrirlos directamente desde el run.
